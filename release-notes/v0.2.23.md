# cc-test v0.2.23

## Workflow protocol tools, phase routing, and confirmation reliability

This release fixes workflow runs that could stop at a phase boundary because the desktop headless agent process did not receive the workflow protocol tool registrations.

### Highlights

- The desktop headless tool pool now registers `submit_phase_completion` and `request_workflow_route` whenever a workflow session binding is present. This restores real workflow completion and server-approved phase routing for development, debug-repair, and feature-extension sessions.
- If a workflow protocol tool is unexpectedly unavailable during a turn, the server refreshes the workflow runtime binding once and sends an internal retry instruction. The user is not asked to type a separate “continue” message for this runtime recovery.
- Confirmation cards now lock immediately after the first click, prevent duplicate transition requests, show a visible pending message, and recover cleanly after stale-state or transition errors.
- Confirmation-card next-step text remains server-first: a server-approved return from Stage 5 or Stage 6 to Stage 4 is displayed as the actual Stage 4 target rather than as a linear Stage 7 transition.
- Explicit `allowedTools` / `disallowedTools` contracts are authoritative over natural-language prerequisite wording. Implementation phases that explicitly allow Write/Edit/Bash no longer lose those tools merely because their instructions mention implementation documents; planning and debug restrictions remain runtime-enforced.

### Verification

- Server workflow protocol, route, phase-permission, and shipped-pack regression suite: 140 passing tests, 893 assertions.
- Desktop workflow UI/state regression suite: 191 passing tests.
- Desktop TypeScript check and production web build passed.
- No workflow ZIP/template files were changed in this release. The fix is in runtime tool registration, runtime recovery, policy enforcement, and desktop transition UI/state.

## Upgrade notes

- Existing workflow sessions keep their persisted workflow state. When a protocol tool was absent in a live run, the runtime can rebind once and retry the structured action without treating the phase as complete.
- Windows installers and the GitHub Release are built remotely by GitHub Actions from tag `v0.2.23`; no local installer artifact is uploaded from this machine.

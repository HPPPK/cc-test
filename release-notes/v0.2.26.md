# cc-test v0.2.26

## Workflow return-route confirmation recovery

This release fixes a workflow state conflict that could leave the confirmation card stuck on “submitting” after a repair loop.

### Fixed

- When Stage 5 or Stage 6 routes back to Stage 4 for repair, a subsequent Stage 4 completion that already targets the normal validation stage no longer creates a second pending route for that same target.
- A normal Stage 4 → Stage 5 continuation now keeps one pending completion as the source of truth, so confirmation completes and the workflow resumes instead of leaving the card permanently pending.
- AskUserQuestion choices that directly continue to the already-pending next phase still auto-confirm and resume that phase; they no longer remain on the prior phase after the duplicate-route safeguard.

### Upgrade notes

- No workflow ZIP/template files changed.
- Existing stuck sessions retain their old persisted state; restart or retry those sessions after upgrading. New repair loops use the corrected route behavior.
- Windows installers and the GitHub Release are built remotely by GitHub Actions from tag `v0.2.26`.
